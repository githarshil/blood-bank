## 2. Tables Created (8 tables + 3 views)

### Core Tables

| Table | Purpose | Records |
|-------|---------|---------|
| **DONOR** | Blood donors with eligibility tracking | 5 sample donors |
| **PATIENT** | Patients requiring transfusions | 3 sample patients |
| **HOSPITAL_STAFF** | Medical staff managing blood bank | 4 sample staff |
| **DONATION** | Donation records with test results | 6 sample donations |
| **BLOOD_INVENTORY** | Blood stock with batch tracking | 8 inventory batches |
| **BLOOD_REQUEST** | Patient blood requirements | 4 sample requests |
| **REQUEST_INVENTORY** | N:M junction for request-inventory | Auto-populated |
| **ALERTS** | System notifications & logs | Auto-generated by triggers |

### Features by Table

#### DONOR Table
- Age constraint: 18-65 years
- Unique phone and email
- Tracks last donation date and count
- Status: Active/Inactive/Suspended
- Indexes: Blood type, Status, City, Combined (blood+status)

#### DONATION Table
- Links to DONOR (1:N via DonorID)
- Blood type validation
- Medical metrics: Hemoglobin, Blood Pressure, Temperature
- Status workflow: Collected → Tested → Approved/Rejected
- Indexes: Donor, Blood type, Status, Date

#### BLOOD_INVENTORY Table
- Batch-level tracking with unique constraint (BloodType, ExpiryDate, StorageLocation)
- Expiry date validation (> today)
- Quantity constraint: >= 0
- Status: Available/Reserved/Expired/Discarded
- Auto-updated LastUpdated timestamp

#### BLOOD_REQUEST Table
- Links to PATIENT (1:N via PatientID)
- Links to HOSPITAL_STAFF (N:1 via StaffID)
- Priority levels: Normal/Urgent/Emergency
- Status workflow: Pending → Approved → Fulfilled
- Date validation: RequiredByDate > RequestDate
- Audit fields: RequestDate, FulfillmentDate

#### REQUEST_INVENTORY Table
- N:M junction mapping requests to specific inventory batches
- Tracks quantity allocated from each batch
- ON DELETE CASCADE for requests, RESTRICT for inventory
- Composite PK prevents duplicates

#### ALERTS Table
- Flexible alert types: Low_Stock, Expiry_Warning, Request_Pending, Donor_Inactive, Blood_Shortage, High_Demand
- Polymorphic design: RelatedEntity + RelatedEntityID
- Status tracking: Active/Resolved/Acknowledged
- Priority: Low/Medium/High/Critical

## 3. Triggers Implemented

### Trigger 1: `trg_add_donation_to_inventory`
**Event:** AFTER UPDATE on DONATION  
**Condition:** When Status changes to 'Approved'

**Function:**
- Automatically adds approved donations to BLOOD_INVENTORY
- Sets expiry date as 45 days from donation date (blood shelf life)
- Uses `INSERT...ON DUPLICATE KEY UPDATE` to add quantity to existing batches
- Updates LastUpdated timestamp

**Business Logic:**
```
Donation Approved → Auto-add to inventory → Makes blood immediately available
```

---

### Trigger 2: `trg_low_stock_alert`
**Event:** AFTER UPDATE on BLOOD_INVENTORY  
**Condition:** Checks both low stock AND expiry warnings

**Functions:**

**Part A - Low Stock Alert:**
- Threshold: < 3 units
- Only inserts if no existing active alert (avoids duplicates)
- Priority: 'Critical' if ≤ 1 unit, 'High' if 1-3 units
- Prevents duplicate alerts for same batch

**Part B - Expiry Warning Alert:**
- Trigger: 7 days before expiry
- Only if Status = 'Available'
- Only if not already alerted
- Tracks days remaining for staff action

**Business Logic:**
```
Inventory Updated → Check thresholds → Auto-generate alerts if needed
```

## 4. Stored Procedure 1: `FulfillRequest(req_id)`

### Purpose
Fulfills a blood request by:
1. Validating request exists and is not already fulfilled
2. Checking sufficient inventory
3. Deducting from inventory (using cursor loop for batch-level tracking)
4. Creating junction records in REQUEST_INVENTORY
5. Marking request as fulfilled
6. Logging to ALERTS

### Transaction Support
- **START TRANSACTION** → All-or-nothing execution
- **ROLLBACK** on error → Maintains data consistency
- **COMMIT** on success

### Key Features

**Validation Steps:**
1. Request exists
2. Status not already fulfilled
3. Sufficient inventory available for blood type
4. Inventory not expired

**Allocation Algorithm:**
- Cursor loops through available inventory sorted by expiry date (FIFO)
- Allocates full batches first, then partial quantities
- Updates inventory after each allocation
- Creates REQUEST_INVENTORY records for audit trail

**Error Handling:**
- Descriptive error messages
- Transaction rollback on failure
- Success/failure status returned to caller

### Usage
```sql
CALL FulfillRequest(1);  -- Fulfill request ID 1
-- Output: SUCCESS message or ERROR message
```

## 6. Cursor Procedure: `RemoveExpiredStock()`

### Purpose
Maintains inventory by:
1. Finding all expired batches
2. Logging each batch to ALERTS
3. Marking inventory as Expired
4. Zeroing quantity

### Processing Flow

**For Each Expired Batch:**
1. Create ALERT record with details:
   - Blood type
   - Quantity removed
   - Expiry date
2. Update BLOOD_INVENTORY:
   - Status → 'Expired'
   - QuantityAvailable → 0
   - LastUpdated → Current timestamp
3. Track statistics

### Cursor Definition
```sql
SELECT InventoryID, BloodType, QuantityAvailable, ExpiryDate
FROM BLOOD_INVENTORY
WHERE ExpiryDate < CURDATE() 
  AND Status IN ('Available', 'Reserved')
```

### Return Value
- Shows number of batches removed
- Shows total quantity removed
- Transaction logs to ALERTS for audit trail

### Usage
```sql
-- Run daily to clean up expired stock
CALL RemoveExpiredStock();

-- Output: SUCCESS message with statistics or "No expired inventory found"
```

## 12. Troubleshooting

### Issue: "Insufficient inventory" error when fulfilling request
**Solution:** Check `vw_blood_stock_summary` for availability. May need new donations or request cancellation.

### Issue: Alerts not appearing after inventory update
**Solution:** Ensure triggers are created (check SHOW TRIGGERS). Verify threshold conditions (< 3 units or ≤ 7 days).

### Issue: Can't delete donor
**Solution:** RESTRICT constraint on DONATION.DonorID. Donors with donation history cannot be deleted (by design).

### Issue: Duplicate alerts for same condition
**Solution:** Triggers check for existing active alerts before inserting. Resolve existing alerts or check alert deduplication logic.
